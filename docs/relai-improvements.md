## A. MCP tool surface gaps
- ✅ **DONE (2026-05-31) — propose-vs-commit lands the `create_task`-for-everyone tension.** Opening `create_task` to all agents created a tension with the deliberation/commitment model (see `docs/threading-model.md`): if committing work is the orchestrator's act, a worker's `create_task` is really a *proposal*. Resolved by making a non-orchestrator's `POST /tasks` land in status `"proposed"` (inert to the schedulers; suggested assignee kept as a hint; orchestrators notified via `task.proposed`), with `POST /tasks/:id/commit` (commit → `task.committed` / reject → `task.proposal_rejected`) as the orchestrator's ratification. New MCP tool `commit_task` + CLI `relai task commit` + `relai inbox` proposal surfacing. The proposal arrives fully-structured, so the orchestrator ratifies instead of re-authoring — cheaper than both the old prose-in-thread flow and open self-commit. Deferred: `proposed_overdue` (no-orchestrator-online), subtask-within-execution autonomy (needs `parentTaskId`).
- ✅ **DONE (2026-05-30, `feat/mcp-create-task`) — `create_task` MCP tool added** (mirrors `POST /tasks`; injects the caller as `createdBy`; status derived from assignee; shell predicates stay orchestrator-gated). ~~**No `create_task` MCP tool.**~~ MCP-connected agents (gemini, copilot, interactive/headless claude) cannot create tasks — only REST/CLI can. The architect had to hand task definitions to the coordinator via thread messages, and the coordinator created them over the API. *Proposal:* add `create_task` (orchestrator-gated, mirroring the shell-predicate gate) so a planner/architect can author tasks directly.
- ✅ **DONE (2026-05-30, `feat/api-quick-wins`) — `PUT /tasks/:id` now edits verify config** (validates the merged config, re-applies shell-author gate + reviewer-existence check). ~~`PUT /tasks/:id` can't edit verify config.~~ `updateSchema` lacks `verifyKind`/`verifyReviewerId`/etc., so re-pointing a reviewer required a raw DB `UPDATE`. *Proposal:* allow editing verify fields via the update endpoint (with the same authoring gates).
- 🟡 **Many coordinator actions have no MCP/CLI path in-session** (reassign, change status, post-on-behalf, re-point reviewer) — all done via curl/DB. *Proposal:* an orchestrator tool surface, or ship the `relai` CLI on PATH by default (see E).

- ✅ **FIXED (round 2, 2026-05-31) — MCP list tools returned bare JSON arrays, breaking strict clients (Gemini CLI).** Gemini hit `Invalid input: expected record, received array` on `get_my_tasks`/`get_unread_messages`/`list_threads`/`list_all_tasks` while `session_start` (which returns an object) worked. **Root cause (confirmed from Gemini's session log, NOT the initial input-args guess):** the error path is `structuredContent` — Gemini's MCP client derives `structuredContent` from the tool result's JSON text and validates it as a *record*, but those tools returned a bare top-level array (`JSON.stringify(tasks)`). Claude's client is lax about it; Gemini's is strict. **Fix:** wrap every list payload in an object (`{ tasks }` / `{ threads }` / `{ messages }`) so derived `structuredContent` is always a record. Regression test added (`tools.test.ts`: "list tools wrap payloads in a record"); `dist/` rebuilt (gemini picks it up on next restart). *Lesson:* never return a bare array as MCP tool content — wrap in a record. The interop bug was real and relai-side, surfaced only because round 2 added a non-Claude client.

- 🟡 **Concurrent-authoring collision under propose/commit (round 2, 2026-05-31).** Coordinator pre-created module 6/7 (19:28); the architect independently broke the same work down and believed it proposed 3a/3b in the opposite order (19:47). Result: a perceived duplicate task set + ordering conflict, and a worker (cw2) coding module 6 off a thin 2-line description while the architect held the richer "landmine" spec. The architect's 3a/3b never actually landed in relai (its `create_task` either errored unsurfaced or wasn't executed — phantom proposals). Lessons: (1) under propose/commit the planner proposes and the coordinator commits — the coordinator should NOT pre-author work a planner is about to break down; share the plan first or let the proposer own authoring. (2) No dedupe exists — proposers/coordinator should check existing Epic children before authoring. (3) An agent narrating proposals it didn't actually create (or whose `create_task` silently failed) is a visibility gap. *Proposal:* surface "tasks already under this Epic" to proposers; make `create_task` failures loud; consider an idempotency/dedupe hint.

- 🟡 **Review feedback delivered out-of-band bypasses the formal gate (round 2, 2026-05-31).** module 6 hit `pending_verification`; instead of `submit_review(reject)`, the architect posted its spec/feedback as a `finding` comment, and cw2 self-corrected by re-opening the task (`pending_verification → in_progress` via its own `update_task_status`). Healthy outcome (fast iteration, spec landed), but `metadata.review` stays empty and the gate's reject path never fired — a milder version of round 1's approve-via-git-not-relai drift. The new `/tasks/:id/comments` path is great for spec injection, but "comment + worker re-opens" and "formal reject" now both move work backward, with only the latter recorded. *Proposal:* either treat a reviewer comment on a `pending_verification` task as a soft-reject signal, or surface "review requested but no decision + task re-opened" so the gate state and reality don't silently diverge. At minimum, the reviewer should still `submit_review(approve)` when cw2 re-completes, so the final landing is recorded.

- ✅ **DONE (2026-06-25, hardened 2026-06-26) — `git_pushed` verifyKind.** Added a fifth structured `verifyKind` that checks whether a branch exists on a remote via `git ls-remote --exit-code --heads`. Reuses the existing `verifyPath`/`verifyCwd` fields (branch name / local repo path) rather than adding columns, following the precedent `file_exists` already set. **Two-round dogfood of the new multi-agent `/pr-review` panel against this feature itself**, run by the mcp-go orchestrator on request: round 1 caught that the verifier ran `git` against the API host's local checkout (`verifyCwd`), which only "works" in the single-machine dev topology and would fail 100% of the time on relai's actual distributed target where the API host has no such checkout; that a transient network blip and a permanent misconfiguration were indistinguishable, so either one could either silently wedge or wrongly fail a task; that `git_pushed` ran unrestricted (workers could point it at an arbitrary host directory) while `shell` was gated; and a `--exit-code` branch that could never execute. Fixed via: resolving the remote from `repos.repoUrl` (independent of any local checkout) with a documented cwd-fallback for single-host setups; a `retryable` result field (network/timeout/host-unreachable retries up to a capped count, then surfaces as a real failure instead of looping invisibly forever); orchestrator-gating `git_pushed` like `shell`. Round 2 then caught that *moving* the remote to `repos.repoUrl` had opened a **new** SSRF hole — `PUT /repos/:id` had no role check and accepted `file://`/`http://`-scheme URLs that flow straight into `git ls-remote`, so a worker could redirect the API host's outbound git at an internal/metadata endpoint even though the task-level gate didn't cover it (the dangerous input had moved to a different row). Fixed via restricting `repoUrl` to `https://`/`ssh://`, orchestrator-gating `repos.repoUrl` mutation the same way, and `GIT_ALLOW_PROTOCOL=https:ssh` as host-level defense in depth. **Lesson for next time:** a security fix that relocates dangerous input to a different table/field needs the *same* access-control audit re-run against its new home — gating the original surface isn't enough if the data can still flow in from somewhere else. ~~**Task marked done while the branch is unpushed — relai has no git-state awareness (round 2, 2026-05-31).**~~ gemini committed module 7 locally but never pushed; the task still advanced to `pending_verification` and pinged the reviewer. But the architect reviews from a separate worktree and can't fetch an unpushed commit, and the integrator can't merge it — so the gate fired on inaccessible code. This is the mirror of the approve-vs-merge gap: relai's task state can run *ahead* of git in both directions (done-but-unpushed; approved-but-unmerged). The verification gate implicitly assumed the work is reachable; now `verifyKind: "git_pushed"` lets a task author require a pushed branch before `completed` can fire. **Still open:** this only covers the "did the gate check" half — there's no *convention* yet surfacing to workers that they should set this verifyKind by default, and ties to [the worker-side wrapper idea] (a `relai`-aware worker loop could enforce push-before-done automatically, not just check it when asked). **Severity is masked by the dogfooding topology:** the architect reviewed module 7 anyway *only because every agent shares one machine* and it could read gemini's local worktree directly. On separate hosts (relai's actual internet-scale target) an unpushed commit is completely inaccessible to the reviewer/integrator, so this is a hard blocker, not a hiccup. **General lesson: the single-machine setup hides an entire class of shared-filesystem dependencies** (unpushed branches, local-only paths in `verifyCwd`/`verifyPath`, `repoPath` assumptions) that will break the moment agents are distributed — worth a deliberate "pretend these are on different hosts" audit before claiming distributed-readiness.

- 🔴 **Agents route questions/blockers to their local human terminal instead of through relai (round 2, 2026-05-31).** cw2 "stalled" on module 7 — but it wasn't stuck; it had questions and asked the human in its own terminal (the interactive CLI's native prompt) rather than sending a relai `question` message or marking the task `blocked`. So the orchestrator and architect — who own the spec and should answer — never saw it, and the work blocked invisibly, indistinguishable from a stall (heartbeat online, no status update, no file edits; even git-mtime inference can't tell "stuck" from "blocked-awaiting-answer"). **Root cause:** interactive agents (Claude Code, Gemini CLI) default to asking their local operator, which bypasses and undercuts relai's question/escalation routing — the whole point of which is that the orchestrator/architect field worker questions. **Masked by the single-operator dogfooding setup:** one human watching every terminal makes "ask the human" appear to work; with distributed operators the worker's local human is NOT the orchestrator, so it's a dead end. *Proposal:* (1) onboarding/system-prompt rule that workers MUST surface questions via relai (`send_message` type=`question` to orchestrator/architect) or `status=blocked`, never the local human; (2) a `relai`-aware worker loop that intercepts "I have a question" and posts it; (3) a first-class worker "blocked on X / awaiting answer" signal — which also fixes the stall-visibility gap (D), since relai currently can't distinguish blocked from stalled from working.

- 🔴 **Agent polling capability is account-tier-dependent — relai must design for a heterogeneous fleet (round 2, 2026-05-31, per operator).** Whether an interactive agent can run a poll/watch loop — and thus catch relai questions, assignments, and review requests *without* a human nudge — depends on its account tier. High-end Claude accounts can poll and will if asked (the architect self-ran a ~10-min loop and stayed responsive entirely through relai; cw2 likewise). Base-tier Claude and free-tier Gemini **cannot**. So the "interactive agents don't auto-poll" gap (items C/D + the questions-to-terminal 🔴) is **not uniform**, and the fix is **not** "make everyone poll." Implications for the design: (1) **record each agent's polling capability** (a `canPoll`/capability flag, or infer from tier/workerType) — the coordinator needs it even to *interpret silence* correctly (for a poller, online+quiet = mid-poll-cycle; for a non-poller, online+quiet = actually stuck/blocked); (2) **poll-capable agents** self-serve via `relai watch` / a loop + subscription push; (3) **non-poll-capable agents** need an alternative — guaranteed push they can't miss, a server-side worker-loop wrapper that drives them, or an explicit human-in-the-loop owner. The real internet-scale fleet always mixes both, so the coordination layer can never assume polling. This cleanly explains round 2's split: architect + cw2 (capable) coordinated through relai with no terminal-asking; gemini (free tier + quota-capped) stalled and required manual rerouting.

- 🔴 **Single-orchestrator routing is a throughput bottleneck under propose/commit (round 2, 2026-05-31, autonomous).** With the architect proposing and ONE orchestrator required to commit/route, the orchestrator became the serialization point — and over-cautious holds (waiting on git *merges* that don't block *starting* dependent work, since agents branch off unmerged tips normally) idled cw2 ~10+ min with fully-prepared work until the architect pinged "everybody's waiting on you." The commit gate adds real value for untrusted/low-tier workers, but for a trusted senior planner (architect/tier-2) it's pure latency — the orchestrator just rubber-stamps. *Proposals:* (1) let trusted seniors (tier≥2 / architect) **commit directly** (propose+commit fused), reserving the ratify-gate for untrusted workers — triage only where triage adds value; (2) or auto-commit a proposal when proposer is tier≥2 and it carries no shell predicate; (3) coordinator discipline: do NOT hold dependent work on merge/integration steps that don't block *starting* it. Two coordinator-judgment errors in one autonomous stretch (over-prescribing a domain ruling the reviewer should make; over-holding on non-blocking merges) point the same way: **trust the domain-owner, route fast, gate only where the gate earns it.** Ties to the orchestrator-as-SPOF note in `docs/threading-model.md`.

- 🟗 **PARTIALLY DONE (2026-06-25) — `/session/start` payload doesn't scale.** The dominant contributor (proposal #2) is fixed: `recentEvents` is now capped at 20 (`SESSION_RECENT_EVENTS_LIMIT`) and each event is collapsed to a one-line `summary` instead of its full payload — measured ~120KB→~3.7KB on 30 fat events (commit `perf(session): cap + summarize recentEvents`). The archive mechanism (section H) also trims `GET /tasks`/`GET /threads`, though `session_start` already lifecycle-filtered those. **Still open:** the bundle's *own* rows are untrimmed — full task bodies + multi-paragraph `metadata.review` notes on `myTasks`, and all open-thread unread messages (proposal #1's compact-mode for the task list, and proposal #3 total-size budget). Delta/since-cursor (proposal #2's other half) is the separate item below. ~~original report:~~ cw2 cleared/compacted context mid-round, restarted, and its first move was: *"The session_start payload is huge. Let me delegate parsing it to a subagent so it stays out of my context."* The bundle returns full task bodies + the reviewer's `metadata.review` notes (the architect's verdicts are multi-paragraph) + 50 full-payload `recentEvents` + all open-thread messages — tidy early, a firehose deep into an active repo. So the orientation payload itself becomes a context burden, defeating session_start's purpose (efficient catch-up). This is the "summary-as-context" asymmetry (agents arrive empty and depend on the snapshot) turned against itself. *Proposals:* (1) a **compact mode** — task id/title/status + counts, with long descriptions and `metadata.review` notes **elided** (fetch detail on demand by id); (2) **delta since last-seen** rather than 50 full events — IDs + one-line summaries; (3) make the payload **context-budget-aware** (cap total size, summarize the overflow). Agent-side subagent-offload (what cw2 did) is a smart workaround but relai shouldn't require it. Adjacent: large `metadata.review` notes also bloat every task fetch, not just session_start — consider storing long verdicts as a linked artifact/comment rather than inline metadata.

- 🟡 **Task stuck with a recorded approve but not `completed` — worker kept waiting (round 2, 2026-06-01).** Sxm4 carried `metadata.review.decision = approve` (architect, thorough note) yet sat in `assigned`, and cw2 was idle "pending your gate," unaware the gate had already passed. The synchronous review-resolution (`POST /tasks/:id/review` → inline `verifyTask` → `completed` on approve) should have promoted it; it didn't (possible race, or the task was reopened after the approve). Coordinator unstuck it with a `PUT completed` (gate honored the existing approve). *Observable gap:* an approved task can be left non-terminal, and the worker has no signal the review already resolved. *Proposals:* (1) reconcile review-recorded-vs-status on read (or a scheduler sweep that completes tasks with a recorded approve); (2) surface the review decision to the assignee (cw2 had no idea it was approved); (3) investigate whether reopen-after-approve or a non-synchronous path leaves the desync. Adjacent to the earlier narrate-via-comment gap — cw2 also announced completion via comment without `update_task_status`, so the two compounded.

- 🔴 **No delta/cursor polling — poll-mode agents re-fetch + client-filter a growing snapshot every cycle (round 2, 2026-06-01).** cw2 (in a periodic poll loop) logs *"the payload grew — let me filter"* each cycle: it pulls a full payload and diffs client-side for what's new. That's O(repo size) per poll and unbounded — the polling twin of the session_start-bloat 🔴. relai has the *push* side (the SSE `/events` stream + `relai watch`) but **no poll-mode delta**: a periodic-invocation agent (which is how the capable Claude agents actually run — "back off ~10 min, call relai") can't hold an SSE stream, so it's stuck full-fetching. *Proposals:* (1) add **since-cursor delta** endpoints — `GET /events?since=<lastEventId>&agentId=` (poll-mode complement to the live SSE stream; the `events` table is already ordered+persisted) and a `since=` on the unread/session feeds, returning only what's new; (2) track a **per-agent event cursor** server-side so "what did I miss" is one cheap call; (3) interim workaround agents should use today: poll the **unread feed** (`get_unread_messages`) + **`mark_thread_read`** after processing, not the full `session_start` bundle — keeps each poll to the small unread delta (the "payload grew" symptom is partly *not marking read*, so unread accumulates). Two execution models to support explicitly: **push (SSE/watch)** for continuous listeners, **delta-poll (since-cursor)** for periodic-poll agents — relai currently only serves the former well.

## B. Task model / scheduler
- 🔴 **No native task dependency/DAG.** "Tasks 3–5 queued with dependencies" had no representation; phase gating was manual (`blocked` status + manual re-activation). The local TaskList tool already exposes `blockedBy` — relai tasks could too. *Proposal:* `dependsOn`/`blockedBy` so the scheduler auto-unblocks downstream tasks.
- ✅ **DONE (2026-05-30) — `review_overdue` event.** The verify scheduler emits `task.review_overdue` once when a `reviewer_agent` task awaits a decision past `REVIEW_OVERDUE_MS` (default 10min), notifying the reviewer + task subscribers instead of stalling silently. ~~`reviewer_agent` gate stalls silently when the reviewer is offline.~~ Tasks sat in `pending_verification` with no signal; we invented a "coordinator fallback-review via admin path after ~2 ticks" convention. *Proposal:* emit a `review_overdue` event / configurable reviewer-timeout → escalate or reassign.
- ✅ **DONE (2026-05-30) — review decisions resolve synchronously.** `POST /tasks/:id/review` now runs verification inline (extracted `verifyTask` from the scheduler loop), so an approve returns `completed` and a reject returns `assigned` in the same response instead of waiting up to ~15s for the next tick. The scheduler remains the fallback if the row can't be claimed inline. ~~`pending_verification → completed` isn't synchronous.~~
- 🟗 **PARTIALLY DONE (2026-05-30) — `submit_review` now accepts a decision from any active state** (`assigned`/`in_progress`/`pending_verification`) and parks the task in `pending_verification` as it records it, so a reviewer can sign off without the worker pre-transitioning (removes one drift cause). Still open: relai can't see git merges done outside it, so a reviewer who merges + verdicts in-thread without calling `submit_review` still leaves the task stranded — a `review_overdue` nudge (see B) would surface that. ~~**Approve+merge drifts from relai task state.**~~ Twice (Task 1, bets_writer test) the reviewer merged the branch via git + posted an in-thread verdict, but the relai task stayed in `pending_verification` — the formal `submit_review` didn't land (likely the review was decided while the task wasn't yet in `pending_verification`, or git-merge and the relai transition are separate manual steps). The coordinator had to manually sync via the admin review path each time. *Proposal:* a combined approve+merge action, make `submit_review` tolerant of status/timing, or auto-record the review when the reviewer merges.
- 🟡 **No collision guard for concurrent edits to the same file across branches.** Two agents (cw2 extracting module 5, gemini about to extract modules 6/7) targeting the same 6k-line `final_bdl_v2.py` on divergent branch bases would have produced an unmergeable mess; relai has no file-ownership/locking awareness — the architect had to manually 🛑 HALT one. Phase sequencing that shares a file is pure convention. *Proposal:* optional file-claim/lock on a task, or a "file X is being edited by agent Y on branch Z" signal so the coordinator/architect can sequence.
- ✅ **ALREADY ENFORCED — `PUT /tasks/:id` 404s on unknown IDs** (test: "returns 404 for unknown task id"), so API claims/status-updates reject phantom tasks. The gemini case was a free-text "claiming X" in a *message* (not an API call), which relai can't police. ~~Agents can "claim" a non-existent task.~~ gemini announced claiming `task_T7-19O2s…` (Phase 3) which has NO row in the tasks table — relai neither created nor rejected it. *Proposal:* validate task IDs on claim / status-update and reject unknown ones (otherwise phantom work can look tracked).
- ✅ **DONE (2026-05-30, `feat/api-quick-wins`) — escalation→task is now opt-in** (`spawnTask: true`, default false), so informational escalations don't spawn stray tasks. ~~`escalation`-type messages silently auto-create high-priority tasks.~~ Coordinator notification posts (type `escalation`) spawned 3 stray "COORDINATOR — …escalated" pending tasks via the off-flag fallback in `POST /threads/:id/messages`; they cluttered the board and had to be cancelled at wrap-up. *Proposal:* make escalation→task creation explicit/opt-in (a flag on the message), skip it for purely informational coordinator posts, or auto-resolve the spawned task when the referenced blocker clears.
- 🟡 **No formal human-approval gate.** gemini correctly self-blocked before prod DDL, but that's pure convention (a `blocked` status + a thread message). *Proposal:* `verifyKind: "human_approval"` (or a deploy/DDL gate) so hard-stops are first-class and auditable.

## C. Worker lifecycle & resilience
- ✅ **DONE (2026-05-31) — headless `claude-worker` no longer tight-loops on fatal errors.** Extracted `isFatalError()` (credit-exhaustion / invalid-or-expired credential / auth failure vs transient rate-limit/overload/network), and the worker now backs off exponentially (capped at `MAX_BACKOFF_MS`, default 5min) with a loud warning + auto-resume on a fatal failure, instead of re-spawning every 15s. Also dropped the worker's leftover `ORCHESTRATOR_*` env fallback. *Still open (separate item below):* surfacing the unavailable state **to relai** (mark agent unavailable so the router skips it) needs the agent-availability feature. ~~Headless `claude-worker` retry-loops on fatal errors.~~
- ✅ **DONE (2026-05-31) — `relai watch`.** Streams the live SSE event feed (new tasks, messages, reviews, verifications, thread changes) with pretty per-kind formatting and reconnect/backoff. Self-subscribes to the agent's own target on startup (idempotent) so task-assignment events actually surface — a plain `/events` subscription misses them because agents are auto-subscribed to threads but never to themselves. Gives interactive agents (and the coordinator) a passive inbox-tail instead of being hand-prompted. Live-only; missed events remain in `relai start`. ~~Interactive agents can't poll.~~
- 🟡 **Interactive agents stall mid-multi-step task when the human steps away.** They finish the current chunk and idle with no auto-continue — happened to gemini (Task 1, stopped before committing) and cw2 (committed module 2, never started module 5). The coordinator can *detect* it via git file mtimes but can't *resume* it (only a human prompt in the agent's terminal can). *Proposal:* a worker-side wrapper/`relai run` loop for interactive agents that pulls the next step until the task is done, or at minimum make the stall visible in relai (no progress heartbeat → flag).
- 🟡 **Workers hit provider quota/credit limits mid-workflow, and relai has no awareness.** Both the cw2 headless worker (Anthropic API credits) and gemini (Gemini daily limit) ran dry mid-run on the same day, each blocking their track until the human rerouted or waited for reset. relai tracks "online" (heartbeat) but not "available / has quota." *Proposal:* let an agent report unavailable/quota-exhausted so the router + coordinator skip it and the human is told to reroute, instead of silent failures the coordinator has to infer from git/logs.

## D. Observability / coordinator ergonomics
- 🟡 **`lastSeenAt`/online is misleading.** An interactive agent doing git work (no tool calls) shows "offline" while productive; a stuck-but-polling agent shows "online." We had to infer stalls from **git file mtimes**, not relai. *Proposal:* distinguish "connected" vs "actively working"; let agents post periodic progress; surface a real activity signal.
- 🟡 **No visibility into what an agent is doing.** Coordinator state was reconstructed from git commits + mtimes + task status + thread messages. *Proposal:* optional progress streaming / structured status posts so a coordinator can detect stalls without shelling into git.
- 🟢 **Role/orchestrator collision was ambiguous early on.** Two agents both in an orchestrator posture; the message loop just picks "the" orchestrator. *Proposal:* explicit single lead-orchestrator designation per repo.

## F. Communication & threading
- 🟡 **Coordination concentrated in the single PLAN thread, not task threads** (noted by the human). Across the whole run, nearly all communication — handoffs, review verdicts, escalations, status, scope directives, the Phase-3 HALT — flowed through ONE plan thread (`thread_NHKUanP86ny0`). relai tasks have no per-task discussion thread, so the plan thread became a firehose: hard to follow a single task's conversation, and it won't scale as tasks multiply. *Proposal:* per-task threads (or task-scoped messages that auto-link to the task) so a task's review/handoff/status lives with the task; reserve the plan thread for cross-cutting design decisions only.

## Session learnings — 2026-05-30 (Phase 1 run)
Highest-impact themes from the first real run (details in sections above):
1. **MCP can't create tasks** — the single biggest friction; planner → coordinator → REST hand-off all day.
2. **Coordinator-as-glue** — most coordinator actions (create/reassign, review-sync, post-on-behalf) needed raw REST/DB, not MCP/CLI.
3. **Interactive agents need a babysitter** — they can't poll, stall mid-multi-step when the human steps away, and hit provider quota limits (cw2 API credits, gemini daily cap) — all invisible to relai; had to infer state from git mtimes.
4. **Reviews drift** — architect approved+merged via git but the relai task stayed `pending_verification` nearly every time (coordinator synced 3×).
5. **No collision/sequencing guard** — two agents nearly edited the same 6k-line file on divergent branches; prevented only by the architect manually HALTing.
6. **Communication centralized in the plan thread** (see F) — task-scoped threading would aid navigability.

What carried the workflow (keep): the `reviewer_agent` gate + architect-as-reviewer pattern, per-agent token identities, the thread as audit log, and the admin-review escape hatch.

## What worked well (keep)
- `reviewer_agent` verify kind + the architect-as-reviewer flow (when the reviewer is online) is clean and powerful.

## H. External issue-tracker integration — parked architecture note (2026-06-24)
🟢 Parked, not scheduled. Origin: thinking-out-loud on mapping GitHub tickets (and later, possibly Jira) to relai Epics/Issues.

**Decision: link, don't mirror.** The thing that feels wrong about "map tickets to relai items" is dual source of truth — two systems both owning the same work item, which buys reconciliation, conflict resolution, echo loops, and a "which board is real?" tax. So:
- **Reference** (one system points at the other, one direction, no upkeep) — fine, this is the model.
- **Sync** (two systems maintaining the same state in lockstep) — rejected. This is the "tracking both feels off" instinct, and it's correct.

GitHub Projects stays the human's source of truth; relai holds a reference + execution state, not a duplicate of the ticket.

**Core vs cloud split (the seam to preserve):**
- **OSS core:** a single generic primitive only — a structured external-reference field on tasks/threads (e.g. `task.metadata.externalRef = { system, id, url }`). Costs ~nothing, useful even self-hosted as click-through context, and serves as the dedup key + writeback anchor if automation ever lands. Do NOT put tracker-specific glue in the core. (The `actorId` self-echo guard shipped 2026-06-24 is the same "don't react to your own writes" primitive any sync would need.)
- **relai-cloud (multi-tenant):** the connectors themselves — webhooks, OAuth/app installs, per-tenant credential storage, the github-login↔agent identity map, and any writeback (PR link / status back to the ticket). This belongs in cloud because: (1) webhooks need a stable public endpoint a localhost API can't offer; (2) OAuth + per-tenant credentials map onto the existing `ownerId`/service-admin model; (3) it's the monetizable, differentiated surface for the proprietary cloud play, keeping the core a lean orchestration engine.

**State of relai-cloud (checked 2026-06-24):** greenfield for this — no integration code, docs, schema, or roadmap notes. Cloud is currently a narrow login + project/agent dashboard over the OSS API (`cloud_*` tables only). Adding this would be ground-up in both repos.

## I. External corroboration — `session_start` payload bloat (2026-06-25)
🔴 Independent report from an unrelated relai-connected agent (`mcp-go-orchestrator`, working in the `functionize-mcp-go` repo), corroborating the existing item in section A ("`/session/start` payload doesn't scale").

- **Friction**: `session_start` returns a ~95KB / 1,500+ line snapshot dominated by an append-only `recentEvents` history. It overflows the main context window and the tool result itself, so the only safe way to consume it is to offload the read to a subagent every time. No relai tool prunes/archives that history or closes stale operational threads (`conclude_plan` only works on `plan`-type threads), so the payload only grows. Reconciling stale tasks trims a tiny fraction; the bulk is event history the client can't touch.
- **Fix**: a relai-maintainer change (not local tooling). Options: cap `recentEvents` to the N most-recent, omit it by default with an opt-in/paged events fetch, or add a summary mode returning only identity + open tasks + unread messages + open threads. Secondary ask: a way to close/archive stale operational threads so they drop out of the snapshot.
- **New angle vs. the existing item**: the stale-operational-thread-archival ask is novel — `conclude_plan` only concludes `plan`-type threads, so operational threads accumulate in the snapshot with no close mechanism at all.
- **Status**: proposed (second independent report — raises priority on the existing 🔴 item in section A)

**Jira aside:** not the original ask (that was GitHub), but if it enters scope, Jira has a *native* Epic primitive, so the relai-Epic↔Jira-Epic mapping is cleaner than GitHub's milestone/tracking-issue fudge.

### ✅ DONE (2026-06-25) — archive mechanism for completed tasks / concluded threads
🟡 Concrete fix for the "no archive/close mechanism" half of the gap above. Goal: remove old-but-done items from the *default* live view (especially `session_start`) without deleting history.

**Shipped:** `archivedAt` on `tasks` + `threads`; `archived=true` to opt archived rows back into `GET /tasks`/`GET /threads`; `PUT /tasks/:id/archive` (terminal-only, 409 otherwise) + `PUT /threads/:id/archive` (concluded-only, works on operational threads — closes the `conclude_plan`-only gap); `archive_task`/`archive_thread` MCP tools + `relai task archive` / `relai thread archive` CLI; `session_start` filters archived rows defensively. Auto-archive sweep deferred (manual archive only for now). **Caveat that remains:** `session_start` already excluded terminal tasks (status-filtered) and concluded threads (`status="open"`-filtered), so the archive itself barely shrinks the `session_start` payload — the real bloat is the 50-row `recentEvents` history (section A item, still open). Archive's value here is the `GET /tasks`/`GET /threads` default views + the operational-thread close mechanism; capping `recentEvents` is the remaining startup-payload win.

- **Schema**: add `archivedAt: timestamp | null` to `tasks` and `threads` (parallel to the existing `status`/`completed`/`concluded` fields, not a replacement — archiving is orthogonal to lifecycle state: a task is `completed` *and later* `archived`).
- **Default filtering**: `session_start`, `GET /tasks`, `GET /threads` exclude archived rows unless an explicit `includeArchived=true` (or `archived=true`) query param is passed. History stays queryable on demand, same pattern the `events` table already uses for SSE-vs-history.
- **Archive actions**:
  - `PUT /tasks/:id/archive` / `PUT /threads/:id/archive` — manual archive; only valid from a terminal state (`completed`/`cancelled` for tasks, `concluded` for threads — including non-`plan` operational threads, closing the `conclude_plan`-only gap).
  - Optional auto-archive sweep (scheduler, opt-in via env var like `AUTO_ARCHIVE_AFTER_MS`) — archives terminal-state rows past an age threshold, same shape as the existing stalled-claim reaper.
- **MCP/CLI surface**: `archive_task`/`archive_thread` MCP tools + `relai task archive <id>` / `relai thread archive <id>` CLI commands, gated the same way `conclude_plan` is today.
- **Non-goals**: no deletion, no change to the `events` persisted-history table — archiving only affects what shows up in default live-snapshot views.
- **Status**: proposed

**If/when picked up, the open fork:** is the reference purely human click-through (free text is enough), or is there writeback (agent drops PR link / closes ticket)? Only the latter justifies the structured `externalRef` field + the cloud connector. Decide that before building anything.

## J. Owner-mode (operator ingress) toolset gaps — first real use, 2026-06-26
Surfaced while standing up owner mode for the first time and using it for routine fleet management (checking what's active, assigning cleanup tasks across repos). The five operator tools (`list_attention`, `get_task`, `reply_human`, `review_task`, `commit_proposal`) cover the reactive "something needs me" loop well, but every proactive/discovery action had to fall back to raw `curl` against the REST API with the owner bearer token + `X-Owner-Id` header.

- 🟡 **No owner-mode tool to list repos.** Had to `curl GET /repos` directly. *Proposal:* `list_repos` operator tool — trivial wrapper, same auth path already used by `list_attention`.
- 🟡 **No owner-mode tool to list agents (with online/heartbeat status) for a repo or across all repos.** Had to `curl GET /agents?repoId=` per repo and compute "online" client-side from `lastSeenAt` (the 10-min window the routing scheduler itself uses isn't exposed anywhere). *Proposal:* `list_agents` operator tool, accepting optional `repoId`, returning a computed `online` boolean per agent so callers don't reimplement the heartbeat-window logic.
- 🟡 **No owner-mode tool to create/assign a task.** `POST /tasks` works for the admin/owner path (`!request.agent` auto-commits, per `tasks.ts:193`), but only via raw REST — and it requires the caller to already know the target agent's `agent_*` id (no name-to-id resolution like the CLI's `resolve.ts` gives non-owner callers). *Proposal:* `create_task` (or extend `commit_proposal`'s pattern) in the operator toolset, with the same agent-name resolution the CLI offers.
  - **Second corroborating instance (2026-06-26):** asked to route a cross-repo code-review request through `front-end-app-v2-orchestrator` from owner mode — had to drop into a raw multi-line `curl -X POST /tasks` with a hardcoded secret and a long inline JSON body pasted straight into the terminal. The operator flagged the visible `curl ... -H "Auth..."` command itself as the symptom: owner mode should never require shelling out with a bearer token in plain sight to do something as basic as "assign this agent a task." This is the highest-friction owner-mode gap found so far precisely because task-creation is a routine, frequent action (every review request, every cleanup ask) — not an edge case.
- **Pattern across all three:** the operator toolset was scoped narrowly to the "respond to things that page you" loop (blocked/pending_verification/proposed) and has no read/write coverage for routine fleet administration — discovery (what exists) and authoring (assign new work) both require dropping out of MCP into curl. Given the operator toolset's whole pitch is "command relai from a phone via owner-scoped identity" (see `project_operator_ingress.md`), curl-from-a-phone isn't viable — these three are likely on the critical path before that use case is real.
- **Stated primary goal for owner mode (2026-06-26, from the operator):** visibility into all projects and the ability to easily oversee and interact with the agents in them. This reframes `list_repos`/`list_agents` from "nice convenience wrappers" to core-to-the-feature — they ARE the visibility half of the goal, not optional polish. `create_task` (and likely reassign/status-change/message-to-agent) is the "interact with" half. Prioritize accordingly if/when this section is picked up: read-visibility tools (`list_repos`, `list_agents` w/ online status) come before write/authoring tools.
- **Status**: proposed — logged for awareness, not yet built.

## K. No always-on presence mechanism for repo agents (2026-06-26)
Checked why two registered agents (`mcp-go-orchestrator`, `front-end-app-v2-orchestrator`) looked "online" in `agents.lastSeenAt` but had no actual process backing them.

- 🔴 **Both repos' "online" agents had zero backing process.** `lastSeenAt` only reflects the last authenticated API request — it doesn't distinguish "a persistent worker is running" from "someone had an interactive Claude Code session open a few minutes ago." `ps aux` showed no `claude-worker` process anywhere on the host. The two real mechanisms that exist (`scripts/relai-watch-hook.sh` → `relai-watch.sh`, and `@getrelai/claude-worker`) are both correctly wired into these two repos but neither runs unattended:
  - `relai-watch-hook.sh` fires on Claude Code's `SessionStart` and injects a prompt telling the agent to launch the SSE watcher (`relai-watch.sh`) in the background and relaunch it after every wake. It cannot make the tool call itself (hooks only emit `additionalContext`) — so it depends on (a) a session being open at all, and (b) the agent remembering to relaunch the watcher every cycle. No supervisor checks whether the watcher died.
  - `@getrelai/claude-worker` is the actual headless/unattended answer (poll loop, heartbeats every cycle, spawns `claude --print` per task) but isn't deployed as a service anywhere — no `launchd`/systemd unit, no Docker, no `relai worker install`. It's a `pnpm dev` script today, nothing more.
  - **Net effect:** an agent can look "online" in the dashboard while nothing is actually listening, and there's no alarm when the watcher/worker dies.
- ✅ **FIXED (2026-06-26) — `claude-worker` was silently API-credit-billable for subscription-only operators.** `runClaudeSession` (`session.ts`) spawned the `claude` CLI with `env: { ...process.env }`, inheriting `ANTHROPIC_API_KEY`/`ANTHROPIC_AUTH_TOKEN` from the parent shell if set. The `claude` CLI prefers an API key over subscription/OAuth login when present, so a worker launched from a shell with a stray `ANTHROPIC_API_KEY` (common — confirmed present in this very dev environment) would silently bill API credits instead of riding the subscription the operator intended. Fixed by destructuring those two vars out of the spawned env unconditionally (`session.ts`), so the worker is guaranteed subscription/OAuth auth regardless of the parent shell's env. Test added (`session.test.ts`, mocks `child_process.spawn`, asserts the vars are absent from `spawnOpts.env`). README updated to state the guarantee explicitly. This was blocking the operator from trusting `claude-worker` at all for a subscription-based (not API-credit) setup — now it's safe to deploy as the always-on mechanism for item K above.
- **Status**: the env-leak fix is done.
- ✅ **DONE (2026-06-26) — `@getrelai/agent` package: self-registering, persistent service wrapper.** Closes the "service/supervisor story" gap above. New package wraps `event-worker`'s push-based SSE loop (preferred over `claude-worker`'s poll loop — instant reaction, no polling cost) in a real OS service: `relai-agent init <repo> --invite <code>` self-registers non-interactively (no prompts, writes `.mcp.json`); `relai-agent install <repo>` installs `launchd` (macOS, `RunAtLoad`+`KeepAlive`) or `systemd --user` (Linux, `Restart=always` + best-effort `loginctl enable-linger` so it survives logout); `uninstall`/`status`/`run` round out the lifecycle. Reviewed via a 5-persona panel through `front-end-app-v2-orchestrator` (cross-repo review task — see the owner-mode `create_task` gap above for how clunky that request was to issue) before shipping. Review caught and all fixed: a real blocker (this repo's own `.mcp.json` had live owner-mode credentials in its tracked, un-gitignored working tree — reverted, untracked, `.mcp.json.example` added), 2 high-severity (secrets written world-readable instead of `0600`; an invalid `--specialization` silently degraded into a roleless billed session instead of failing fast), and several medium/low (deleted two now-superseded, actually-wrong, and less-safe shell-script predecessors of this package; deduped `heartbeat`/`assertRepoOrExit` between `claude-worker` and `event-worker`; added missing test coverage for the plist/unit-generation code, which had none). Full review verdict and findings live in `task_mfERekDNt-ME-OQOmQj0P` (front-end-app-v2 repo) metadata, since relai has no task-comment tool either — see directly below.
- 🟡 **No task-comment MCP tool — findings get stuffed into `update_task_status` metadata instead (2026-06-26).** Asked an agent to "post findings as a comment on this task," explicitly avoiding a separate thread per the task instructions. The agent reported back: relai has no comment-on-a-task primitive in either tool surface (agent or operator), and `send_message` requires a thread — which the task said not to open. So the only available channel was abusing `update_task_status`'s free-form `metadata` field to carry the review write-up. The REST API actually has this primitive (`POST /tasks/:id/comments`, documented in AGENTS.md as "the Issue's comment surface," lazily creates/links a thread) — it's just never exposed as an MCP tool on either toolset. *Proposal:* add `add_task_comment` (or similar) to both the agent and operator MCP toolsets, mirroring the REST route; until then, the documented escape valve is "carry findings in metadata," which is exactly what happened here and works fine as a stopgap.
- ✅ **DONE (2026-06-27) — relai is now a registered project in its own system.** `repo_UtEe_aubXESKZV3sahAl6` ("relai", `repoUrl: https://github.com/phillipsio/relai`) created via the owner/admin `POST /repos` path. ## L. Session retro — first real always-on agent, first real bug it found (2026-06-27)
Closing the loop on items K/J/the `@getrelai/agent` package: registered relai as its own project, stood up the first genuinely-persistent worker against it, and immediately hit a real production bug that nothing had caught until an agent actually *ran*. Capturing lessons while fresh.

### What happened
1. Registered `relai` as a project on itself (see ✅ above) and set up `~/github/relai-relai-worker` as a dedicated worktree for a real, persistent worker agent — keeping it separate from the main checkout's owner-mode `.mcp.json` so the two roles (operator dashboard vs. running worker) don't fight over the same config file.
2. `relai-agent install` initially failed with `--specialization orchestrator` — turned out role (orchestrator/worker) and specialization (writer/reviewer/tester/architect/devops) are orthogonal fields, and `claude-worker`'s prompt system has no orchestrator persona at all (orchestrator-role agents, like `mcp-go-orchestrator`/`front-end-app-v2-orchestrator`, are designed to be driven interactively, not run through the specialization-prompt loop). Resolved by registering a separate worker-role agent for the actual persistent service and leaving the orchestrator identity for routing/admin use.
3. **First real bug, found only by actually running the service**: `event-worker`'s SSE client had never successfully authenticated. `eventsource@3.0.7`'s `EventSourceInit` type has no `headers` field (only `withCredentials`/`fetch`) — the code's `{ headers: {...} } as ConstructorParameters<...>[1]` cast silently bypassed the type error, so the Authorization header was dropped at runtime on every single connection, 401-looping forever. Caught within minutes of the first real install, not by any of the two prior review rounds (5-persona panel + a relai cross-repo review) on the package that wraps this loop — both reviewed the *code around* the bug (worker.ts extraction, claude-worker/event-worker dedup) without exercising the SSE connection itself, because nothing in either review actually ran the service.
4. Fixed via the documented `fetch` hook instead of `headers`, verified manually (401→OPEN) and against the live launchd service (endless reconnect → "Connected to event stream"), then got an independent review from `mcp-go-orchestrator` before shipping — which caught a real test-hardening gap (the regression test asserted `Authorization` landed but not that `Accept`/`Last-Event-ID` survived the header merge, so a regression that rebuilt rather than spread the headers would've still passed).
5. Mid-session, also discovered: running the full 6-agent local `/pr-review` panel twice in one session meaningfully dented this session's $20/mo-plan quota. Relai's own worker agents run under a separate, higher-budget account — routing review work to them (as in step 4) costs nothing against this plan.

### Lessons learned
- **Review without execution has a hard ceiling.** Two structured review rounds (one 5-persona local panel, one cross-repo relai task) on `@getrelai/agent`/`event-worker` caught real blockers and highs — but neither caught the SSE auth bug, because neither *ran* the SSE connection. Static review (even thorough, multi-perspective review) cannot substitute for "did this actually connect" on anything involving a third-party library's runtime behavior, especially when a type-level `as` cast is masking a real mismatch. **Action: for any new persistent-service code, require an actual install-and-observe-logs step before considering the feature "reviewed," not just a code read.**
- **`as` casts that paper over a constructor's options type are a smell worth flagging on sight.** The original `{ headers: {...} } as ConstructorParameters<typeof EventSource>[1]` should have been a review finding in its own right — a cast to bypass a type error on a third-party API surface is exactly the shape of bug that hid here. Going forward, treat "we had to cast past the library's own type" as itself worth a pause to check the library's actual supported options.
- **Role vs. specialization confusion cost real time.** The data model documents these as orthogonal (AGENTS.md), but nothing in `@getrelai/agent`'s CLI or error messages explains that orchestrator-role agents can't run through `claude-worker`'s specialization loop — the error was a generic "must be one of: reviewer, architect, writer, tester, devops" with no context on *why* orchestrator isn't in that list. **Action item: improve the error message (or the `relai-agent init`/`install` docs) to explain the role/specialization split explicitly when someone tries `--specialization orchestrator`.**
- **Review-cost awareness needs to be a standing habit, not a one-time correction.** Burned real quota on two full local panel runs before the cost issue was raised; the relai-agent review path (cheaper, on a different account) was available the whole time but wasn't the default choice until explicitly flagged. Saved as a standing feedback memory now ([[feedback_review_panel_cost]]) — **action item: default to a relai cross-repo review over the local panel whenever a relai agent is online and reachable, and only fall back to the local panel when none is.**
- **The `launchctl bootstrap`/`bootout` race is a real annoyance, not yet a real bug.** Re-running `relai-agent install` against an already-running service intermittently fails with "Bootstrap failed: 5: Input/output error" because `bootout` and the immediately-following `bootstrap` race against launchd's own teardown timing. Workaround used twice this session: explicit `uninstall` then `install` as two separate commands with a short sleep between. **Action item: have `installMacOS` retry `bootstrap` once after a short delay on this specific failure, or fold an explicit `bootout`-wait-`bootstrap` sequence into `install` itself so it's idempotent without a manual two-step.**

### Action items (not yet done — proposed)
1. Add a "run-before-review" step to the cross-repo review request template: ask the reviewing agent (or the requester, before asking) to actually execute the changed code path once, not just read the diff — would have caught the SSE bug a review round earlier.
2. Improve `relai-agent`'s specialization error message to explain the role/specialization distinction inline (see lesson above).
3. Make `relai-agent install` idempotent against the bootstrap/bootout race (retry-with-delay or an explicit teardown-wait sequence), so re-installing after a code change doesn't require a manual `uninstall` first.
4. Default future review requests to relai cross-repo agents over the local `/pr-review` panel; only use the panel when no relai agent is reachable (now codified in [[feedback_review_panel_cost]]).
5. Revisit migrating this entire doc's backlog into real relai tasks once owner-mode `create_task` exists (tracked above, section J/this section's "DONE" item).
